## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| Use root-level planning files plus antigravity working-memory files | Keeps the project resumable across sessions without losing the user-requested format |
| Keep backend and frontend scaffolds intentionally thin in Phase 1 | Preserves flexibility for later phases while still making verification possible |
| Use an in-process TTL cache for fetched data in Phase 2 | Good fit for a single FastAPI process and keeps the implementation small while reducing repeated yfinance calls |
| Keep `/data/market-data/{stock_id}` focused on close-price signals plus separate quarterly fundamentals | Matches the spec's track separation and keeps downstream signal processing decisions explicit |
| Feed Phase 3 transforms from a dedicated cached close-price loader | Reuses the existing fetch/cache layer while keeping FFT and spectrogram code clean and stateless |
| Return PNG by default from `/signal/spectrogram/{stock_id}` but also support `?format=json` | Satisfies the Phase 3 verification target while preparing the later D3-based frontend heatmap work |
| Add an explicit `hht` config section to `stocks.json` | Keeps all transform parameters config-driven, including `max_imfs` and `frequency_bins` |
| Centralize shared convolutional blocks and regression-head construction inside `BaseModel` | Keeps model variants consistent and reduces drift between per-stock and unified architectures |
| Let `ModelRegistry` resolve canonical and fallback artifact filenames for the embedding model | Absorbs an ambiguity in the spec without leaking file-naming assumptions into later route code |
| Split Phase 5 datasets by `label_timestamp`, not just window-start order | The model predicts the price at `t + prediction_horizon_days`, so the split boundary should track the true target time to avoid subtle leakage |
| Build training notebooks from Python-authored cell payloads rendered through a thin Jinja2 notebook shell | This keeps the templates valid JSON while still making notebook content config-driven and easy to validate before route wiring |
| Add an environment-driven smoke mode to the generated notebooks for verification | Allows end-to-end notebook execution checks without changing the emitted notebook files or affecting normal Colab behavior |
| Persist per-stock retraining reports in `backend/models/model_store/reports/` and prediction histories in `backend/retraining/prediction_history/` | Keeps retraining artifacts aligned with the existing model store while giving drift detection a stable on-disk source of truth |
| Use `asyncio.to_thread()` inside `RetrainWorker.retrain()` | Prevents the synchronous PyTorch training loop from blocking the FastAPI event loop while still keeping the route/scheduler APIs simple |
| Resolve retraining mode to `per_stock` whenever that mode is configured | Matches the stock-scoped API and avoids retraining the shared unified artifact multiple times for each stock-triggered run under the current `model_mode: "both"` config |
| Serve generated notebooks via `FileResponse` from the route layer instead of returning raw JSON text | Preserves browser download semantics while still reusing the already-verified generator output on disk |
| Introduce route-level Pydantic response models for the newly hardened model and training endpoints | Makes the contract explicit in FastAPI while keeping the JSON shapes consistent for frontend work that follows |
| Back the training page endpoints with an in-memory runtime/event state and a background thread | Allows `POST /training/start` to return immediately and `GET /training/progress` to emit per-epoch SSE updates without blocking the request thread |
| Let `/model/predict/{stock_id}` fall back to a trained per-stock checkpoint when the configured unified predictor is missing | Keeps the API useful under the current artifact set and makes the fallback explicit in the response so it does not silently mask missing unified weights |
| Terminate `/live/stream/{stock_id}` after one SSE message when the exchange is closed | Gives the frontend a reliable closed-market snapshot and countdown without keeping an idle EventSource connection open for hours |
| Centralize unknown-stock and unknown-exchange response shaping in the shared route utilities | Extends the structured-error contract across multiple endpoint families with one change instead of re-implementing it in each route module |
| Type the remaining non-streaming FastAPI JSON responses explicitly and annotate `responses=` maps as `dict[int | str, ...]` | Keeps OpenAPI accurate while satisfying mypy's stricter FastAPI decorator expectations |
| Keep `backend/training/notebook_cells.py` under a file-level flake8 ignore | The file intentionally stores embedded notebook code blocks whose natural line lengths are part of the generated source, so broad reflow would hurt readability more than it would help maintainability |
| Use a Vite/TypeScript alias for the root shared config file instead of a copied frontend JSON file | Makes the frontend config source explicit and removes the risk of silent drift between frontend and backend settings |
| Drive the route shell from one shared manifest and add a dedicated mobile nav component | Keeps the desktop and mobile navigation surfaces aligned while preserving lazy page loading |
| Implement the first stock-detail charts with a small shared SVG chart system and a typed market-data hook | Delivers live backend data quickly without introducing a chart dependency that the repo does not already use |
| Introduce `d3` specifically for the signal-analysis page rather than replacing the lightweight SVG primitives used by the stock-detail page | Limits dependency growth while still using the right tool for viridis scales, tick generation, and dense heatmap coordinate math |
| Downsample the frontend spectrogram preview grid before rendering SVG cells | Keeps the CWT/HHT views responsive in the browser while preserving the exact backend response for future drill-down or export workflows |
| Keep the live testing screen on `/live` and drive stock changes through a `?stock=` query param | Preserves one canonical route for the page while still allowing deep links to a selected stock snapshot |
| Separate `useMarketStatus` from `useLiveMarket` on the frontend | Lets the page show an immediate exchange-status baseline even when the SSE stream closes after a single after-hours snapshot |
| Keep the How It Works screen on `/explainer` and drive stock changes through a `?stock=` query param | Preserves one canonical explainer route while still allowing the animation to follow a chosen stock |
| Reconstruct the full close-price signal from overlapping STFT frames for the Phase 13 explainer | Lets the raw-signal and sliding-window panels use the same backend payload as the spectrogram and FFT panels instead of fetching a second market-data source |
| Derive the explainer's prediction panel from the current STFT frame plus the next reconstructed signal point | Keeps the educational animation self-contained on `/signal/stft-frames/{stock_id}` while still showing predicted-vs-actual context |
